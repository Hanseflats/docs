Authenticated control plane for **every** Playbooks audience play — not just competitor reviews. Same endpoints for YC companies, new hires, lookalikes, Google Ads audiences, Substack, Upwork, and the rest of the dashboard catalog.

Discover plays live with `GET /api/v1/audiences/plays`, then pass any `play_slug` to validate/launch. Inputs differ per play — always read `required_inputs` from `GET /api/v1/audiences/plays/:slug` before launching.

## Auth

```text
Authorization: Bearer YOUR_API_KEY
```

or `x-api-key: YOUR_API_KEY`.

## Idempotency-Key

Optional header on `POST /api/v1/audiences` (launch only). Safe retries — not a play input.

```text
Idempotency-Key: my-agent-run-123
```

|  |  |
| --- | --- |
| **What it does** | If the same API key user launches again with the **same** key, the API returns the existing audience instead of starting a duplicate run. Response `status` is `"idempotent_replay"`. |
| **When to send it** | Agents, scripts, or webhooks that may retry on timeout/network errors. |
| **What value to use** | Any stable string you choose (UUID, `{play}-run-1`, etc.). Values like `accelerator-alumni-companies-1` in curl examples are placeholders, not special API tokens. |
| **New run** | Use a **new** key (or omit the header) when you intentionally want another audience. |

Omit the header if you do not care about duplicate launches on retry.

## Endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/v1/audiences/plays` | Catalog of active plays |
| `GET` | `/api/v1/audiences/plays/:slug` | Play \+ `required_inputs` schema |
| `POST` | `/api/v1/audiences/validate` | Dry-run param validation |
| `POST` | `/api/v1/audiences` | Launch audience |
| `GET` | `/api/v1/audiences` | List your audiences |
| `GET` | `/api/v1/audiences/:id` | Detail \+ recent leads/companies |
| `PATCH` | `/api/v1/audiences/:id` | Update cadence / delivery / title |
| `POST` | `/api/v1/audiences/:id/sync` | Push to webhook / Instantly |
| `GET` | `/api/v1/audiences/:id/export` | CSV (`?type=leads\|companies`) |

JSON uses **snake\_case**. Volume target is top-level `number_of_leads` (default `100`). Play-specific inputs go in `source_params`.

## Credits

Charged after successful finds (not at launch):

| Result | Credits |
| --- | --- |
| Company found (no personas) | 1 |
| Company found \+ matched buyer persona | 2 |
| Person-first play (reviews, Substack, Upwork, new hires, etc.) | 1 per person |

## Destinations

Set delivery targets on launch or later via `PATCH`. Lens auto-sync and manual `POST .../sync` use the same destinations.

| Field | Purpose |
| --- | --- |
| `delivery_webhook_url` | HTTPS URL. One POST per lead or company row (see [Webhooks](/guides/webhooks)). |
| `delivery_instantly_campaign_id` | Instantly campaign ID (bulk push; leads with email). |
| `delivery_instantly_list_id` | Instantly list ID (bulk push; leads with email). |

Webhook delivery is **one row per request**. Instantly stays bulk.

## Example flow

```bash
export API_KEY=YOUR_API_KEY
export BASE=https://app.lens.ly

# 1) List plays
curl -s "$BASE/api/v1/audiences/plays" \
  -H "Authorization: Bearer $API_KEY" | jq '.plays[] | {slug, name}'

# 2) Inspect a play
curl -s "$BASE/api/v1/audiences/plays/competitor-reviews" \
  -H "Authorization: Bearer $API_KEY" | jq '.play.required_inputs'

# 3) Launch with a webhook destination
curl -s -X POST "$BASE/api/v1/audiences" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: reviews-demo-1" \
  -d '{
    "play_slug": "competitor-reviews",
    "source_params": {
      "websites": "hubspot.com",
      "max_rating": "3",
      "find_work_email": "true"
    },
    "number_of_leads": 100,
    "title": "HubSpot reviewer leads",
    "delivery_webhook_url": "https://webhook.site/your-id"
  }' | jq

# 4) Poll
curl -s "$BASE/api/v1/audiences/AUDIENCE_ID" \
  -H "Authorization: Bearer $API_KEY" \
  | jq '.audience | {id, apify_status, leads_total}'

# 5) Manual sync (optional; auto-sync also runs from the pipeline)
curl -s -X POST "$BASE/api/v1/audiences/AUDIENCE_ID/sync" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{}' | jq

# 6) Export
curl -s "$BASE/api/v1/audiences/AUDIENCE_ID/export?type=leads" \
  -H "Authorization: Bearer $API_KEY" -o leads.csv
```

## Launch / PATCH body fields

```json
{
  "play_slug": "competitor-reviews",
  "source_params": {},
  "number_of_leads": 100,
  "title": "Optional title",
  "cadence": null,
  "delivery_webhook_url": "https://example.com/hooks/lens",
  "delivery_instantly_campaign_id": null,
  "delivery_instantly_list_id": null
}
```

`PATCH /api/v1/audiences/:id` accepts the same destination fields plus `title` and `cadence`.

## Sync

```bash
curl -s -X POST "$BASE/api/v1/audiences/AUDIENCE_ID/sync" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "force_all": false }'
```

- Default: only rows newer than the last successful sync watermark.
- `force_all: true`: re-deliver from the start of the audience (use carefully).

## Webhook payload (one row per POST)

Lead audiences:

```json
{
  "audience_id": "...",
  "play_slug": "competitor-reviews",
  "title": "HubSpot reviewer leads",
  "type": "leads",
  "lead": {
    "id": "...",
    "email": "alex@acme.com",
    "first_name": "Alex",
    "last_name": "Lee",
    "full_name": "Alex Lee",
    "job_title": "VP Marketing",
    "linkedin_url": "https://www.linkedin.com/in/...",
    "company_domain": "acme.com",
    "company_name": "Acme",
    "source_vars": {},
    "created_at": "2026-07-18T12:00:00.000Z"
  },
  "synced_at": "2026-07-18T12:01:00.000Z"
}
```

Company audiences use `"type": "companies"` and a top-level `"company"` object instead of `"lead"`.

Return **2xx** so the row counts as delivered. Partial failures advance the watermark only through the contiguous success prefix (sorted by `created_at`).

## Launch response

```json
{
  "audience": {
    "id": "...",
    "title": "HubSpot reviewer leads",
    "play_slug": "competitor-reviews",
    "apify_status": "pending",
    "cadence": null
  },
  "status": "triggered"
}
```

`status` is `"triggered"` on a new launch, or `"idempotent_replay"` when the same user already launched with that [Idempotency-Key](/api/audiences#idempotency-key).
